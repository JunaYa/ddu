# 加固 DDU macOS 版到「日常可靠」

## Summary

一次聚焦的 macOS 质量加固 pass：把日常会踩、以及让人不敢信任的点修扎实——真实不可逆打码、关闭两个路径穿越漏洞、消除关键路径上的 panic、保存路径不再被启动覆盖、快捷键改了即生效、复制后有反馈、预览浮窗行为更直觉——并新增一个复用现有编辑器的「多张截图拼成一张」功能。目标门槛是「自用 / 日常可靠」。

---

## Problem Frame

项目当前的状态是一个**能用的 macOS 个人工具，套着一份跨平台全能产品的规格**。`docs/` 和 `specs/` 描绘了对标 CleanShot X / ShareX 的能力（滚动截图、云上传、录屏、多平台 OCR），而一次性「实现全部 7 份规格」的提交在 macOS 上确实跑得起来，但留下了几类直接伤害日常信任的缺口：

- **打码是假的**：模糊/像素化只是在画布上盖一层半透明灰色矩形，原始像素仍然完整地留在导出图里。隐私规格（specs/04-ocr-privacy-ai.md）把「默认不可逆打码」列为核心承诺——当前实现直接违背它，是信任级别的缺陷，不是普通 bug。
- **两个真实安全漏洞**：删除历史和导出图片的命令直接接受前端传入的任意路径、不做校验，理论上可删/读/写系统任意文件。
- **日常摩擦点同时是真 bug**：用户在设置里选的保存路径每次重启被覆盖回默认目录；在设置里改的快捷键只存进了 store、全局快捷键仍按硬编码默认值注册（改了不生效）；复制到剪贴板后没有任何反馈。
- **健壮性脆弱**：关键路径上有 ~40 处 `unwrap()`/`expect()`，磁盘满/无权限等常见情况会直接 panic；带标注的复制会不断产生不清理的临时文件；部分设置项（历史最大容量）UI 有但根本没写入存储。

这些缺口让工具「看着全、用着虚」。本轮不追广度，只把已经承诺、且每天会碰到的东西做真、做对、做稳。

---

## Key Decisions

- **门槛是「自用 / 日常可靠」**：质量与信任优先于广度。不做 Gatekeeper 公证打包、i18n、首次引导、自动更新——这些按门槛排除。
- **打码做真，不做视觉占位**：模糊/像素化必须对底层位图区域做不可逆处理，导出后无法恢复原内容。这是本轮唯一一个从「修 bug」跨进「实现功能」的点，因为门槛明确要求「打码真实有效」。
- **拼图复用现有资产**：入口走「历史多选 → 进现有 Fabric 编辑器拼接 → 走现有导出/复制路径」，不新建独立拼图器。MVP 只做横排/竖排两种布局。
- **维持 macOS-only 与现有实现风格**：本轮不动跨平台抽象、不重写依赖 `screencapture` 的捕获引擎。
- **选区/预览浮窗只做行为级修复**：在原生捕获路线内做力所能及的体验改进，不重写成带放大镜/键盘微调的自定义 overlay（那属于广度方向）。

---

## Requirements

**安全与信任**

- R1. 真实打码：模糊与像素化必须对所覆盖区域的底层图像做不可逆的像素处理，导出图中该区域无法恢复出原始内容。当前的画布半透明矩形覆盖（视觉占位）不满足此要求。
- R2. 路径校验：所有接受前端传入文件路径的命令（删除历史、导出图片）必须把目标限制在 app 受控目录内，拒绝越界或包含穿越段（如 `../`）的路径。

**健壮性（不崩溃 / 不丢数据）**

- R3. 消除 panic：捕获、保存、剪贴板、OCR 正则编译等关键路径上的 `unwrap()`/`expect()` 替换为正常错误传播；失败时返回可见错误提示而非进程崩溃。
- R4. 临时文件清理：带标注「复制」产生的临时文件用后即清，不随使用次数无界累积。
- R5. 设置真实持久化：设置界面展示的每一项（含历史最大容量等当前未接上的项）都真正写入存储并在重启后生效；接不上的要么接上、要么从 UI 移除。

**日常工作流**

- R6. 保存路径不再被覆盖：启动时不得无条件把 `screenshot_path` 写回默认目录；用户配置的保存路径在重启后保留，新截图保存到该处。
- R7. 快捷键改了即生效：在设置里修改快捷键后，全局快捷键按新值重新注册；若平台能力要求重启才能生效，则在 UI 中明确告知，不得静默失效。
- R8. 复制后有反馈：复制到剪贴板成功后给出即时可见反馈（toast / 系统通知 / 视觉确认）。
- R9. 预览浮窗行为直觉化：评估当前用 minimize/unminimize 代替 show/hide 带来的副作用（残留在程序坞/切换器、跟随 Space 等），必要时改为真正的显示/隐藏。
- R10. 选区体验小改进：在不重写为自定义 overlay 的前提下，对原生选区流程做力所能及的改进；若原生路线下无可改空间，则明确记录为已知限制而非留作无声痛点。

**拼图（多张合成）**

- R11. 多选合成：用户可从主窗口历史列表中多选若干截图，合并为一张图。
- R12. 拼接布局：MVP 支持横排与竖排两种布局。
- R13. 复用编辑器输出：合成结果进入现有 Fabric 编辑器，可继续标注，并通过现有导出/复制路径输出。

---

## Key Flows

- F1. 历史多选拼图
  - **Trigger:** 用户在主窗口历史列表多选 ≥2 张截图，触发「拼成一张」。
  - **Steps:** 选择布局（横排 / 竖排）→ 应用把所选图按布局合成到一张画布 → 画布进入现有编辑器 → 用户可标注 → 通过现有「另存为 / 复制」输出。
  - **Outcome:** 得到一张可标注、可导出/复制的合成图。
  - **Covered by:** R11, R12, R13

## Acceptance Examples

- AE1. 打码不可逆
  - **Covers R1.**
  - **Given** 一张含敏感文字的截图，**When** 用像素化工具覆盖该区域并导出 PNG，**Then** 重新打开导出的 PNG 并放大，原文字无法辨认或恢复。
- AE2. 路径穿越被拒
  - **Covers R2.**
  - **Given** 调用方向删除/导出命令传入受控目录之外的路径（如 `../../<系统文件>`），**When** 命令执行，**Then** 命令拒绝并返回错误，受控目录外的文件不受影响。
- AE3. 保存路径持久
  - **Covers R6.**
  - **Given** 用户在设置里把保存路径改为 `~/Pictures/DDU`，**When** 退出并重启应用，**Then** 保存路径仍为 `~/Pictures/DDU`，新截图保存到该处。
- AE4. 快捷键生效
  - **Covers R7.**
  - **Given** 用户把区域截图快捷键从 `Cmd+Shift+S` 改为 `Cmd+Shift+1`，**When** 保存设置后按 `Cmd+Shift+1`，**Then** 触发区域截图，且旧快捷键不再触发。
- AE5. 复制反馈
  - **Covers R8.**
  - **Given** 用户在编辑器点「复制」，**When** 复制完成，**Then** 出现可见反馈确认已复制到剪贴板。

## Success Criteria

- 捕获 / 保存 / 复制 / 导出路径在常见失败场景（磁盘满、无权限、路径不存在）下不 panic，并给出可见错误。
- 对导出图重新检查时，打码区域不可逆（可人工或脚本验证无法还原）。
- 已知的两个路径穿越点关闭，越界路径被拒。
- 「展示即生效」：设置界面上每个开关/输入都有真实读写，并跨重启保留。
- ce-plan 能据本文档直接拆任务，无需再追问产品行为。

## Scope Boundaries

**本方向、以后可做（Deferred）**

- 前后对比 / 自由拼贴布局；「截图时连续抓多张再拼」的交互（本轮拼图仅历史多选 + 横/竖排）。
- 带放大镜、键盘微调、尺寸输入的自定义选区 overlay（specs/01 的 CAP-002/CAP-003 完整版）。
- 低优先级 cosmetics（datetime 不补零、Toggle 不随 prop 同步、清理未使用的 demo 组件 `Greet.vue`/`Screenshot.vue`）——顺手则修，不专门追，不阻塞本轮。

**别的方向、本轮不碰（Out of this round）**

- 跨平台（Linux / Windows）落地——当前 Windows 无法编译、Linux 部分损坏，属独立方向。
- 云上传/分享（S3/WebDAV/FTP）、滚动截图、录屏增强（音频/摄像头/裁剪/GIF 完善）、跨平台真本地 OCR。
- 公证打包、i18n、首次引导、自动更新（按「自用」门槛排除）。

---

## Dependencies / Assumptions

- 真实打码预计用 Rust 端图像处理（`image` crate 已在依赖中）或前端 canvas 像素处理；难点在于编辑器里打码是矢量对象，导出时需对底层位图真正生效——具体方案在 planning 决定。
- 拼图复用已在用的 Fabric.js。
- 维持现有对 macOS `screencapture` 的依赖。
- 假设原生 `screencapture -i` 选区 UI 的可改造空间有限，R10 的实际改进幅度待 planning 探索确认。

---

## Outstanding Questions

**Deferred to Planning**

- 打码真实化放后端（`image` crate 处理区域）还是前端（canvas 像素化后烧进位图再导出）——需评估「矢量标注对象 → 导出位图」这一步如何让打码对底层像素生效。
- 快捷键重注册的触发时机（保存即热重载 vs 重启生效）——取决于 `tauri-plugin-global-shortcut` 的能力。
- 选区体验在原生路线下能改到什么程度——planning 探索后定，可能转为「记录限制」。
